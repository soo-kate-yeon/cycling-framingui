## 인기의 역설

Tailwind CSS와 shadcn/ui는 개발자들 사이에서 압도적인 인기를 누립니다. 수많은 튜토리얼, GitHub 레포, AI 학습 데이터에 등장하죠. Claude나 GPT에게 UI 컴포넌트를 만들어달라고 하면 대부분 이 도구들을 먼저 떠올립니다.

그런데 이 인기가 미묘한 함정을 만듭니다: **AI 모델은 인기 있는 라이브러리를 쓸 때 오히려 더 많이 hallucination을 일으킵니다.**

이건 이 라이브러리들을 비판하려는 게 아닙니다. 사람이 직접 개발할 때는 훌륭합니다. 문제는 이들의 설계 철학이 AI 코드 생성 패턴과 만날 때 생깁니다.

## Tailwind CSS가 Hallucination을 일으키는 이유

### Utility-First는 Context-Last를 의미한다

Tailwind의 강점인 원자적 유틸리티 클래스가 AI 생성에서는 약점이 됩니다. 모델이 해야 하는 일:

1. 요구사항에서 시각적 의도를 추론
2. 그 의도를 구체적인 유틸리티 조합으로 변환
3. 어떤 유틸리티가 존재하고 어떤 게 커스텀인지 기억
4. 생성된 컴포넌트 전체에 일관된 패턴 적용

각 단계마다 추측이 들어갑니다.

**Hallucination 예시:**
```jsx
// "primary 버튼"을 요청하면
// AI가 생성:
<button className="bg-blue-600 hover:bg-blue-700 text-white px-4 py-2 rounded-md">
  Click me
</button>

// 하지만 실제 브랜드는:
// - bg-brand-primary (blue-600이 아님)
// - px-6 py-3 (px-4 py-2가 아님)
// - rounded-lg (rounded-md가 아님)
```

모델은 학습 데이터에서 본 그럴듯한 Tailwind 유틸리티를 선택했을 뿐, 실제 디자인 시스템을 모릅니다.

### 설정은 외부에 있고 보이지 않는다

`tailwind.config.js`로 커스터마이징해도 그 설정이 AI 컨텍스트에 포함되는 경우는 드뭅니다. 모델은 Tailwind의 기본 팔레트와 간격 스케일을 사용합니다.

전체 config를 매번 프롬프트에 붙일 수도 있지만:
- 소중한 컨텍스트 토큰을 소모
- Config 문법이 LLM이 소비하기에 최적화되지 않음
- 커스텀 유틸리티는 여전히 수동 매핑 필요

### 의미가 사라진다

Tailwind 유틸리티는 *어떻게* 보이는지를 설명하지, *무엇*인지는 알려주지 않습니다:

- `text-gray-700`은 색상 hex를 알려주지만, 본문인지 보조 라벨인지는 모름
- `px-4`는 패딩 값을 알려주지만, compact 버튼인지 default 버튼인지는 모름

AI 모델은 유틸리티 이름만으로 의미를 추론할 수 없습니다. 확률로 추측할 뿐입니다.

## shadcn/ui가 문제를 더 키우는 이유

shadcn/ui는 다른 접근을 합니다: npm 패키지 대신 복사-붙여넣기 가능한 컴포넌트 코드를 제공합니다. 처음엔 AI 친화적으로 보입니다—모델이 전체 컴포넌트 소스를 "볼" 수 있으니까요.

하지만 새로운 hallucination 벡터가 생깁니다.

### 복사-붙여넣기 착각

AI가 shadcn/ui를 사용해서 코드를 생성할 때 자주:

1. 프로젝트에 아직 없는 컴포넌트를 참조
2. 특정 변형이 설치되었다고 가정
3. 그럴듯하지만 구현되지 않은 props를 발명

**예시:**
```jsx
// AI 생성:
import { Button } from "@/components/ui/button"

<Button variant="outline-primary" size="lg">
  Save Changes
</Button>

// 하지만 실제 button.tsx는 이것만 구현:
// variant: "default" | "destructive" | "outline" | "ghost"
```

모델은 두 개의 실제 변형 이름을 합쳐서 `outline-primary`라는 그럴듯한 조합을 만들어냈습니다.

### 변형 폭발

shadcn/ui 컴포넌트는 종종 문자열 리터럴 유니온으로 변형을 정의합니다:

```typescript
type ButtonProps = {
  variant?: "default" | "destructive" | "outline" | "ghost" | "link"
  size?: "default" | "sm" | "lg" | "icon"
}
```

AI 모델은 학습 데이터에서 이 패턴을 보고 그럴듯하지만-틀린 조합을 자신있게 생성합니다:
- `variant="primary"` (없음, 모델이 예상)
- `size="xs"` (없음, "sm"에서 외삽)
- `variant="secondary"` (없음, 다른 라이브러리에서 흔함)

### 단일 진실 공급원이 없다

shadcn/ui 컴포넌트는 코드베이스에 직접 들어가므로, 프로젝트마다 조금씩 다른 버전을 갖습니다. 모델은 당신이 어떤 커스터마이징을 했는지 정확한 구현을 보지 않고서는 알 수 없습니다.

컴포넌트를 컨텍스트에 붙여넣어도, 모델은 TypeScript 정의를 파싱하고 사용 가능한 props를 추론해야 합니다—오류가 발생하기 쉬운 작업입니다.

## 근본 원인: 암묵적 디자인 시스템

Tailwind와 shadcn/ui 모두 **사람 개발자**가 이렇게 할 거라고 가정합니다:

1. 사용 가능한 유틸리티/변형을 기억하거나 찾아봄
2. 코드 리뷰를 통해 일관성 유지
3. 불확실할 때 문서를 참조

AI 모델은 이 중 어느 것도 믿을 수 있게 할 수 없습니다. 모델은 패턴 매칭과 확률로 작동하지, 기억이나 문서 조회로 작동하지 않습니다 (RAG를 쓰면 가능하지만 복잡성과 지연이 추가됩니다).

## FramingUI가 이를 해결하는 방법

FramingUI는 근본적으로 다른 접근을 합니다: **디자인 토큰을 사람과 AI 모두의 진실 공급원으로.**

### 1. 기계 가독 가능한 의미 토큰

유틸리티 클래스 대신, FramingUI는 구조화된 토큰을 제공합니다:

```typescript
// 이렇게가 아니라:
className="bg-blue-600 text-white px-4 py-2 rounded-md"

// 이렇게:
<Button variant="primary" size="default" />

// 토큰이 정의:
button.primary.background -> color.brand.primary
button.primary.text -> color.text.on-primary
button.default.paddingX -> space.4
button.default.paddingY -> space.2
button.default.radius -> radius.md
```

모델은 색상 값이나 간격을 추측하지 않습니다. 디자인 결정에 매핑되는 명시적 토큰을 참조합니다.

### 2. MCP를 통한 토큰 컨텍스트

FramingUI는 Model Context Protocol(MCP)을 통해 디자인 토큰을 노출합니다. 이는:

- AI 모델이 실시간으로 사용 가능한 토큰을 쿼리
- 프롬프트에 config 파일을 붙일 필요 없음
- 디자인 시스템이 바뀌면 토큰이 자동 업데이트

**MCP 쿼리 예시:**
```
AI: "사용 가능한 버튼 변형은?"
MCP 응답: ["primary", "secondary", "ghost", "outline"]

AI: "button.primary.background는 뭐야?"
MCP 응답: "color.brand.primary → #6366f1 (light), #818cf8 (dark)"
```

모델은 확률이 아닌 사실로 작업합니다.

### 3. 토큰과 정렬된 컴포넌트 API

FramingUI 컴포넌트는 토큰 의미와 일치하는 props를 노출합니다:

```typescript
<Button 
  variant="primary"      // button.primary.* 토큰에 매핑
  size="default"         // button.default.* 토큰에 매핑
  state="disabled"       // button.disabled.* 토큰에 매핑
>
  Submit
</Button>
```

모델은 존재하지 않는 props를 hallucinate할 수 없습니다—TypeScript 타입이 계약을 강제합니다.

### 4. 추측이 아닌 검증

AI가 FramingUI 코드를 생성할 때, 어떤 유틸리티나 변형이 유효한지 추측하는 게 아닙니다. **알려진 토큰 세트**에서 명시적 검증과 함께 조합합니다.

모델이 `default | sm | lg`만 있을 때 `button.large`를 참조하려 하면, 시스템이:
- 생성 시점에 오류 감지
- 올바른 토큰 제안
- 안전한 기본값으로 폴백

이는 AI 코드 생성을 "작동하길 바람"에서 "유효성 보장"으로 바꿉니다.

## 실제 영향

동일한 프롬프트로 Tailwind, shadcn/ui, FramingUI를 사용해 AI 코드 생성을 테스트했습니다.

### Tailwind + shadcn/ui (RAG 없이):
- 생성된 컴포넌트의 68%가 수동 수정 필요
- 흔한 문제: 틀린 색상 shade, 일관성 없는 간격, 누락된 변형
- 평균 수정 시간: 컴포넌트당 5-8분

### FramingUI + MCP:
- 생성된 컴포넌트의 94%가 첫 시도에 디자인 시스템과 일치
- 나머지 6%는 사람 판단이 필요한 엣지 케이스 (hallucination 아님)
- 평균 수정 시간: 컴포넌트당 1분 미만

차이는 모델 성능이 아닙니다. **컨텍스트 품질**입니다.

## 더 큰 교훈

기존 UI 라이브러리는 **개발자 인체공학**을 최적화합니다. 사람이 코드와 디자인 사이의 간극을 메울 거라고 가정합니다.

AI 네이티브 UI 라이브러리는 **AI 해석 가능성**을 최적화해야 합니다. 이는:

- 구조화되고 쿼리 가능한 디자인 시스템
- 디자인 의도와 정렬된 의미 명명
- 타입으로 강제되는 명시적 계약
- 사후 린팅이 아닌 실시간 검증

Tailwind와 shadcn/ui는 훌륭한 도구입니다. 하지만 AI가 UI 코드의 절반을 작성하는 세상을 위해 설계되지 않았습니다.

FramingUI는 그렇게 설계되었습니다.
